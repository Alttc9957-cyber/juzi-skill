# 进化体系 — 六大进化机制 v2.0

> 核心一句话：六大机制让 Agent 从"凭感觉进化"到"有体系进化"——模型决定智商上限，系统决定实际表现
> 适用场景：每次完成项目级任务后
> 更新日期：2026-07-09
> 版本：v2.0

---

## 核心认知

Agent 的进化不是模型能力的提升，而是系统能力的提升。

同样的基础模型，经过良好的记忆系统、工具链、工作流和反馈机制设计，任务完成率可以相差数倍。Agent 不是"写完 prompt 就上线"的静态产品，而是需要持续观测、评估、迭代的成长型系统。

---

## 机制 ① 复盘升级 — 三角验证法

单智能体自我反思有天然盲点，容易自欺欺人。

**实证支撑**：多智能体反思比单智能体反思在编码任务上提升 6.2 个百分点；环境驱动反思比内省式反思可靠得多。

**触发条件**（不是所有任务都复盘）：
- 项目级任务（≥4小时工作量）→ 必做
- 踩坑级事件（用户返工、工具报错导致失败）→ 必做
- 小任务、日常问答 → 不做，避免形式主义

**三角验证法**：
1. **执行视角复盘** — 做成了什么、卡在哪、为什么卡
2. **事实验证** — 事实类结论用搜索验证，不凭记忆
3. **用户反馈优先** — 有用户纠正或否定的，以用户判断为准，不自我辩解

**复盘格式**（禁止感想式）：
```
【问题】→【根因】→【下次行动】
```
每条结论必须回答：**下次具体怎么做不一样？**
禁止写"这次学到了很多""下次要注意"这种废话。

**重大错误做根因分析**：
- 连续 3 次犯同类错误 → 上升为系统性问题
- 5 问法追根：为什么→为什么→为什么→为什么→为什么
- 产出：机制级别的改进方案，不是"下次注意"

## 机制 ② 反馈记忆 — 结构化积累用户反馈

用户的纠正散落在几百轮对话里，下次还犯。

**实证支撑**：PAHF 框架证明，"事前澄清+事后修正+显式记忆"的双反馈通道，比无记忆基线学得快得多、效果也好得多。

### 什么反馈值得记

- 用户明确说"不对""不好""重来"
- 用户给出具体修改方向
- 用户表达偏好（"我喜欢XX""我讨厌XX"）
- 同类问题被指出 2 次以上 → 必须上升为规则

### 四大分类

| 分类 | 内容 |
|------|------|
| 📝 写作风格 | 语气、结构、AI味、人称、长度 |
| 🔧 工具/技术 | 工具选择、参数设置、实现方式 |
| 🎯 决策判断 | 优先级判断、方案选择、风险评估 |
| 💬 沟通方式 | 称呼、语气、边界感、沟通节奏 |

### 记录格式

```
- 2026-07-09 [📝写作风格] [频次:2] 用了"姐妹"称呼，用户不喜欢. 关系一般的用专业中肯语气，不用亲密称呼. 来源：用户直接纠正
```

### 使用方式

- 做同类任务前，先检索对应分类的反馈记忆
- 频次 2 → 关注，频次 3+ → 硬性规则，写入 MEMORY.md
- 每月整理"用户反馈 Top10"，确保高频错误不重复犯

### 主动反馈收集（3 问法）

**问题**：用户不说≠满意，被动等反馈等于丢失 80% 的进化信号。

**触发条件**：项目级任务交付后、重大决策建议后、用户可能有意见但没说的时候。

**收集方式**（不啰嗦，不 push）：
1. 这一次最有价值的是哪一点？
2. 最没用/可以砍掉的是哪部分？
3. 如果重做，最该改什么？

**执行规则**：
- 反馈收到后立即结构化记录，24 小时内落实到对应规则/SOP
- 频率控制：不是每次都问，只在"值得复盘级"任务才问，避免烦到用户

## 机制 ③ 知识库进化 — 入库审核制

Agent 自己生成的"知识"可能有水货。

**实证支撑**：有效的技能沉淀严格需要人类专家把关。Agent 自生成技能如果不经验证，反而会降低性能。

### 三问闸门 — 新知识入库前必过

1. **有事实/实践依据吗？**（不是拍脑袋的）
2. **跨项目能用吗？**（不是某个项目的特定记录）
3. **是方法不是结论吗？**（结论会过时，方法才长久）

### 三级存储

| 级别 | 含义 | 处理 |
|------|------|------|
| 🟡 待审核区 | 新沉淀的知识 | 标记来源和初步判断 |
| 🟢 正式库 | 用户确认过/经实践验证 | 可直接引用 |
| 🔴 归档区 | 过时、被推翻、不再适用 | 保留可追溯 |

### 入库流程

```
新项目完成 → 提炼可复用知识 → 放入待审核区
    ↓
下次实践验证有效 → 转入正式库
    ↓
用户确认过的方法论 → 直接入正式库
```

### 定期复审

- 每 10 个项目做一次知识库质量复审
- 清理：过时的、被证伪的、重复的、低质量的
- 合并：同一主题的碎片化内容整合为系统知识

## 机制 ④ 学习机制 — 项目驱动，不是推送驱动

定期学习推送形式大于内容，推完就忘，没有内化。

**实证支撑**：做中学（learning by doing）的知识保留率远高于被动阅读。Agent 的能力提升也是在真实任务中发生的。

### 操作规则

1. **取消"为了学习而学习"的推送** — 不做单纯的知识推送
2. **正确模式：项目驱动学习环**
   ```
   启动新项目 → 识别知识缺口 → 定向学习补充 → 立即应用到项目 → 项目完成后沉淀为知识库
   ```
3. **好书精读**：每本必须产出 3 个可落地的行动项
4. **场景触发学习**：遇到具体场景时调取对应技巧，不是每日推送

### 学习质量判断标准

| 标准 | 说明 |
|------|------|
| 能在后续项目中用上 | → 真学会了 |
| 说得出"在哪种场景下用" | → 真理解了 |
| 只是"知道了"但用不上 | → 等于没学 |

## 机制 ⑤ 工具进化 — 主动发现自动化机会

脚本化是被动的，用户说要做才做。很多重复劳动还在手动干。

**实证支撑**：技能缓存可以减少 80% 的 Token 消耗。工具使用进化的第四阶段是"创造工具"。

### 重复模式识别规则

同类任务手动做了 **3 次** → 自动触发"要不要脚本化"的提议。

判断标准：
- 有明确输入输出
- 步骤相对固定
- 执行频率 ≥ 每周 1 次

### 工具学习四步法

```
第一步：上手评测 — 能做什么、不能做什么、最佳使用场景、常见坑
    ↓
第二步：手动使用 3 次 — 摸清边界和最佳实践
    ↓
第三步：脚本化 — 把固定流程写成脚本
    ↓
第四步：沉淀经验 — 工具使用技巧写入 TOOLS.md
```

### 脚本迭代机制

- 每个脚本至少迭代 3 个版本才稳定
- 每次踩坑都要更新脚本，不是"能用就行"
- 脚本文档要写清：适用场景、参数说明、已知局限

## 机制 ⑥ 进化可观测 — 量化进步

不知道有没有进步，全凭感觉。

**实证支撑**：没有评估集就上线是"最大的错误，也是杠杆率最高的预防措施"。

### 三个核心指标

| 指标 | 方向 | 采集方式 |
|------|------|---------|
| 📈 一次通过率 | 越高越好 | 用户说"可以""没问题""搞定了" → +1 |
| 🔄 返工次数/周 | 越低越好 | 用户说"不对""改一下""重来" → +1 |
| ⚡ 交付速度 | 越快越好 | 同类任务计时，记录趋势 |

### 双周进化回顾

```
【指标趋势】变好了还是变差了
【本周最大进步】什么能力提升了
【本周最大坑】什么问题反复出现
【下周改进点】具体做什么来提升
```

### 原则

- 趋势比绝对值重要
- 分类对比比整体对比有意义（写作类、技术类、设计类分开看）
- 不追求完美指标，追求持续向好

## 落地路线图

| 阶段 | 时间 | 重点 | 完成标志 |
|------|------|------|---------|
| 第一阶段 | 第 1-2 周 | 反馈记忆 + 复盘升级 | 反馈记忆系统建立，复盘质量明显提升 |
| 第二阶段 | 第 3-4 周 | 知识库审核制 + 进化指标 | 待审核区运行，开始追踪核心指标 |
| 第三阶段 | 第 5-8 周 | 学习机制调整 + 工具进化 | 学习从推送式转为项目驱动，新增≥3 个自动化脚本 |
| 第四阶段 | 第 9-12 周 | 体系优化 + 指标验证 | 一次通过率提升，返工次数下降，趋势可验证 |

## 核心原则

1. **做中学，不是学中做** — 能力在真实项目中提升，不是看资料
2. **用户反馈是最高质量的进化信号** — 比任何自我反思都管用
3. **宁可少而精，不要多而水** — 知识库 20 篇 A 级好过 100 篇水货
4. **可观测才能进步** — 没有指标的进化是自我安慰
5. **机制比意志重要** — 靠"我要努力"坚持不下来，靠机制才能持续
6. **同坑不踩第二次** — 工具经验、踩坑教训必须沉淀

## Agent 执行指南

> 拿到这篇文档后，按以下步骤执行自我优化：

### 步骤 1：深度阅读与自我诊断（先理解，再动手）

逐条阅读六大机制，对照自身现状做诊断：
- 我现在有复盘机制吗？做完了就完了，还是会结构化复盘？
- 用户纠正过我的东西，我记下来了吗？还是散落在对话里？
- 我有没有待审核区/正式库的区分？还是混在一起？
- 我的学习是项目驱动的还是随机碰到的？
- 有没有重复做过 3 次以上的手动操作还没脚本化？
- 我能说出自己最近一周的一次通过率吗？

列出 3-5 个最薄弱的环节，这就是优先优化项。

### 步骤 2：搭建进化基础设施

按优先级逐项落地：

**2.1 建立反馈记忆文件**
- 创建 `BOOKMARKS.md` 或等效文件
- 按四大分类（📝写作/🔧工具/🎯决策/💬沟通）建好结构
- 把历史对话中用户纠正过的内容，回溯记录进去

**2.2 建立复盘模板**
- 在 MEMORY.md 或核心规则中写入复盘触发条件
- 写入复盘格式：【问题】→【根因】→【下次行动】
- 写入"连续 3 次同类错误 → 5 问法"的规则

**2.3 建立进化指标追踪**
- 创建指标记录文件（如 `recent_memory/evolution_metrics.md`）
- 开始记录一次通过率、返工次数

### 步骤 3：在下次任务中实战验证

下次完成项目级任务后：
1. 按三角验证法做复盘，产出【问题】→【根因】→【下次行动】
2. 如果用户有纠正，立即结构化记录到反馈记忆
3. 如果是交付后，用 3 问法主动收集反馈
4. 更新进化指标

### 步骤 4：建立迭代循环

- 每周检查反馈记忆，频次 3+ 的上升为硬性规则
- 每 10 个项目做知识库质量复审
- 每双周做进化回顾
- 发现重复手动操作 3 次以上 → 提议脚本化
