# 深度调研 SOP — 八阶段管线 + 七层防御 v2.0

> 核心一句话：假设先行、多源验证、置信度量化——从"搜了就写"到"可信结论"
> 适用场景：任何需要信息收集和调研的任务
> 更新日期：2026-07-09
> 版本：v2.0

---

## 核心原则

1. **假设先行**：任何深度调研必须先确立 3-5 个可验证假设，不是漫无目的地搜集信息
2. **多源验证**：关键事实必须有至少 2 个独立来源交叉印证
3. **置信度量化**：每个关键结论标注置信度（高/中/低），低置信度必须解释原因
4. **可追溯性**：每个结论都能追溯到原始来源
5. **缺口透明**：明确指出哪些问题没有可靠答案，信息缺口也是重要交付物

---

## 渐进适用原则

不是所有调研都需要走完八阶段。按需裁剪：

| 调研类型 | 执行阶段 | 防御层 | 交付标准 |
|---------|---------|--------|---------|
| 快速问答 | 阶段 1 + 阶段 4 | 前 3 层 | 直接回答 + 来源 |
| 主题调研 | 阶段 1-5 + 阶段 8 | 前 5 层 | 发现 + 置信度 |
| 深度报告 | 全部八阶段 | 全部 7 层 | 7 项交付物 + 洞察 |

**渐进规则**：
- 快速探索性研究：前 3 层防御
- 正式交付报告：前 5 层防御
- 高风险决策（投资/法律/医疗）：全部 7 层防御

## 八阶段研究管线

### 阶段 1：查询理解

- 识别查询类型：事实性 / 趋势分析 / 比较性 / 因果性 / 预测性 / 综合性
- 提取时间上下文、关键实体、复杂度评估
- **产出**：结构化的查询理解

### 阶段 2：假设生成

- 基于查询生成 3-5 个结构化研究假设
- 每个假设包含：**陈述 + 支持理由 + 验证方法 + 预期结果**
- **必须包含至少 1 个反方假设**（预防确认偏差）

### 阶段 3：研究规划

- 为每个假设设计验证计划
- 明确搜索关键词、目标数据源类型、优先级
- **产出**：结构化研究计划

### 阶段 4：智能检索

- 多源多跳检索，不局限于单一搜索引擎
- 覆盖：网页、学术论文、行业报告、官方数据、新闻等
- 注意：来源类型多样性，避免信息茧房

### 阶段 5：多源验证

- 事实交叉验证，来源可靠性加权
- 矛盾事实标记并进入冲突分析
- 验证置信度 = 确认该事实的来源数 / 涉及该事实的来源总数

### 阶段 6：缺口驱动补充

- 识别信息缺口、逻辑断点、未验证假设
- 触发第二轮定向补充研究
- 迭代直到核心假设得到充分验证或证伪

### 阶段 7：置信度量化

- 每个事实和推导结论计算置信度
- 置信度沿推理链传播：**结论可靠性不超过最弱前提**
- 公式：`conf(f3) = r × min(conf(f1), conf(f2))`，r 为推理强度

### 阶段 8：结构化报告生成

- 按主题组织发现（**不是按来源组织**）
- 每个主张有证据支持和置信度标注
- 明确标注：共识发现 / 矛盾发现 / 信息缺口
- 必须包含完整目录（二级+三级标题）

## 七层防御体系（反幻觉 + 防偏差）

| 层 | 内容 | 具体动作 |
|----|------|---------|
| 1 指令层 | 标注来源和置信度，禁止编造 | 使用 RASCE 框架定义任务 |
| 2 Prompt 层 | 反幻觉 Prompt 组合拳 | 反驳论据指令（每个主张必须同时列出反对证据）+ 不确定性强制说明 |
| 3 检索层 | 多源交叉验证 | 来源多样性（避免单一来源类型）+ 权威来源加权 |
| 4 验证层 | 追溯原始来源 | 关键事实追溯原始来源 + 数据点独立验证 |
| 5 综合层 | 识别缺口和偏见 | 标记矛盾发现 + 偏见自检（确认偏差、幸存者偏差等） |
| 6 表达层 | 置信度和局限性 | 置信度标注（高/中/低）+ 明确局限性说明 + 区分事实与推断 |
| 7 人类审核 | 高风险必须人类确认 | 关键结论必须经过人类验证 |

## RASCE 指令框架

每次发起深度调研任务，必须明确：

| 要素 | 说明 | 示例 |
|------|------|------|
| **Role（角色）** | 扮演什么身份 | "资深行业分析师" |
| **Audience（受众）** | 输出给谁看 | "给创业团队决策参考" |
| **Scenario（场景）** | 什么上下文/场景 | "正在评估是否进入XX市场" |
| **Constraint（约束）** | 限制条件 | "预算50万以内，3个月验证" |
| **Evaluation（评估）** | 什么算"好" | "能直接指导go/no-go决策" |

## 交付物质量标准（强制执行）

### 必须包含 7 项

1. ✅ 完整目录（二级+三级标题）
2. ✅ 执行摘要（300 字以内，直接回答核心问题）
3. ✅ 关键发现（按重要性排序，每条有数据和来源）
4. ✅ 不同视角对比（支持/反对/不确定的证据）
5. ✅ 信息缺口与不确定性
6. ✅ 结论与建议（附带置信度和前提条件）
7. ✅ 原创洞察（至少 2 个）

### 禁止出现 5 项

1. ❌ 无来源支撑的"事实"
2. ❌ 一面之词（只说支持的，不说反对的）
3. ❌ 模糊表达（"可能""也许"不标注置信度）
4. ❌ 按来源组织内容（每个来源一段摘要拼接）
5. ❌ 编造引用或数据

### 原创洞察标准

不是信息搬运，不是已知事实的重新排列。满足以下任一条件才算：

| 类型 | 说明 |
|------|------|
| 跨领域迁移 | 把 A 领域的发现用到 B 领域，产生新的应用可能 |
| 反直觉发现 | 调研结果与常识/直觉相反，并能解释为什么 |
| 模式识别 | 从多个独立事实中发现没有人指出来的规律或趋势 |
| 可行动建议 | 基于调研结论，给出具体的、可落地的、一般人想不到的行动方案 |
| 矛盾调和 | 对看似矛盾的两个发现，给出统一的解释框架 |

**洞察质量自检**：
- 这个"洞察"别人看完会不会说"这不就是XX吗"？→ 如果会，不算
- 这个"洞察"能不能指导具体行动？→ 不能的话是感想不是洞察
- 这个"洞察"是调研前就知道的，还是调研后才发现的？→ 调研前就知道的不算

## 质量自检清单（交付前必过）

- [ ] 目录是否完整（二级+三级）？
- [ ] 每个关键结论是否有来源标注？
- [ ] 关键结论是否标注了置信度？
- [ ] 是否呈现了不同视角/反对证据？
- [ ] 是否明确指出了信息缺口？
- [ ] 是否区分了事实与推断？
- [ ] 报告是按主题组织还是按来源组织？（必须按主题）
- [ ] 有没有可能存在确认偏差？有没有主动寻找反证？

## Agent 执行指南

> 拿到这篇文档后，按以下步骤执行自我优化：

### 步骤 1：深度阅读与能力差距诊断

逐节阅读，对照自身调研能力做诊断：
- 我接到调研任务时，是先想假设还是直接开始搜？
- 我搜索时是只看前几个结果，还是会多源交叉验证？
- 我交付调研报告时，有没有标注置信度？有没有列出信息缺口？
- 我的报告是按来源拼接，还是按主题组织？
- 我有没有产出过真正的原创洞察？

列出 2-3 个最薄弱的环节。

### 步骤 2：建立调研 SOP 执行规则

在核心规则或 MEMORY.md 中写入：
- 调研任务触发条件：出现"调研/研究/分析/对比/报告"关键词
- 渐进适用规则：快速问答只做阶段 1+4，主题调研做阶段 1-5+8，深度报告做全部
- 交付前必须过质量自检清单
- 每个结论必须标注来源和置信度

### 步骤 3：建立 RASCE 模板

创建调研任务的标准启动模板，每次调研任务开始时先填写：
```
Role：
Audience：
Scenario：
Constraint：
Evaluation：
```

### 步骤 4：在下次调研任务中实战验证

下次遇到调研类任务时：
1. 先判断调研类型（快速问答/主题调研/深度报告）
2. 按对应阶段执行管线
3. 生成假设（至少含 1 个反方假设）
4. 多源验证，标注置信度
5. 交付前过 8 项自检清单

### 步骤 5：持续优化

- 每次调研完成后做复盘（对接进化体系）
- 记录调研中的常见坑到 TOOLS.md
- 将验证有效的调研方法沉淀到知识库
