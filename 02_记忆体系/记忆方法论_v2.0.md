# 记忆体系 — 三层架构 + 生命周期管理 v2.0

> 核心一句话：不是记得越多越好，而是重要的不能丢，无用的别占位置——分层记忆，重点突出
> 适用场景：每次写入或检索记忆时
> 更新日期：2026-07-09
> 版本：v2.0

---

## 一、记忆体系总览

### 三层架构

| 层级 | 存储位置 | 存什么 | 访问方式 | 容量限制 |
|------|---------|--------|---------|---------|
| 即时层 | MEMORY.md / USER.md / SOUL.md / TOOLS.md / SECRET.md | 核心规则、状态锚点、用户画像、身份定义、工具经验、敏感凭证 | 每次对话自动加载，零延迟 | ≤5KB/文件 |
| 近中期层 | recent_memory/ 目录（index.json + 分类文件） | 项目进度、重要决策、待办事项、方法论、详细规范 | 先读索引定位 → 再读具体文件 | 无上限，按主题分类 |
| 长期层 | 所有历史对话记录、所有文档内容 | 全量历史对话 + 所有文件 | memory_search 语义检索（像搜索引擎） | 全量历史 |

### 检索黄金顺序

1. **先查即时层** — 已自动加载，0 成本
2. **再查近中期层** — 读 index.json 定位，再读文件
3. **最后查长期层** — 语义检索，精确召回

**禁止越级查询。能用前两层解决的就不碰第三层，节省 token。**

## 二、即时层管理

### 五份核心文件分工

| 文件 | 记什么 | 判断标准 | 上限 |
|------|--------|---------|------|
| USER.md | 用户画像 | 换一个全新项目，这条信息还成立吗？成立就归 USER | 2KB |
| MEMORY.md | 规则 + 状态锚点 + 指针 | 要遵守什么 + 现在到哪了，详情写近中期层 | 5KB |
| SOUL.md | 身份定义 | 主人对身份、性格、行为风格的设定 | 5KB |
| TOOLS.md | 工具操作经验 | 能填进"用XX工具时要注意YY"就归 TOOLS | 5KB |
| SECRET.md | 敏感凭证 | API Key、密码、Token | 5KB |

### MEMORY.md 格式规范

两大固定分区：
- **长期行为规则** — 必须遵守的铁律
- **核心状态锚点** — 当前进行中的项目、关键指针

每条格式：`- **粗体短标题**：内容`

指针用代码行块包裹：
```
> 详见 recent_memory/project/项目X.md
```

### 即时层瘦身原则

接近上限时主动精简：
1. **规则类**：保留核心原则，详细操作下沉到知识库/近中期层
2. **状态类**：只留指针+一句话摘要，详情放文件
3. **项目类**：完成的项目归档近中期层，只留当前在途的
4. **经验类**：TOOLS.md 满了就把低频工具的经验下沉

## 三、近中期层管理

### 目录结构

```
recent_memory/
├── index.json          # 所有记忆单元的目录（摘要+标签）
├── project/            # 项目进度快照
│   ├── 项目A.md
│   └── ...
├── decision/           # 重要决策记录
│   └── ...
└── todo/               # 待办事项
    └── ...
```

### index.json 索引机制

每个记忆单元一条索引：
```json
{
  "title": "文件名",
  "summary": "一句话说明内容",
  "tags": ["项目", "方法论"],
  "path": "recent_memory/project/xxx.md",
  "updated": "2026-07-09"
}
```

### 写入规则

什么内容放近中期层：
1. 说不清、需要详细记录的 — 方法论文档、完整 SOP
2. 项目相关的 — 进度、方案、复盘
3. 重要决策 — 为什么选 A 不选 B，决策依据
4. 待办清单 — 中长期的任务列表

更新规则：
- 修改已有文件用 edit_file，不要全量覆盖
- 新建文件用 write_file
- **写入后必须同步更新 index.json**

## 四、长期层管理

### 语义检索机制

- **工具**：memory_search
- **原理**：RAG 向量检索，按语义相似度召回
- **可调参数**：
  - `focus`：messages_only / prefer_messages / balanced / prefer_files / files_only
  - `channel`：coze / feishu / email
  - `max_results`：最多返回条数
  - `start_time / end_time`：时间范围

### 什么时候需要搜长期记忆

1. 用户说"好好想想""你不记得了吗"
2. 涉及历史决策、之前讨论过的方案
3. 不确定某个信息是否讨论过
4. 需要找参考案例或历史经验

## 五、记忆写入判断标准

### 写入前必问三问

1. **换个项目还成立吗？** → 成立 → USER/SOUL/TOOLS（即时层）
2. **以后会复用吗？** → 会 → 近中期层（项目/决策/方法论）
3. **只是过程信息？** → 是 → 不存

### 绝对不写的

- 临时中间结果、草稿
- 一次性的搜索结果
- 普通任务执行过程
- 闲聊内容

### 立即写的（不可延迟）

- 用户说"记住""记一下""以后别忘了"
- 发现新的工具坑/经验（同个坑不能踩第二次）
- 重要决策结论
- 用户偏好、习惯、禁忌

## 六、上下文分层管理

### 三层信息分类法

对话中的每条信息，先判断层级：

| 层级 | 标记 | 定义 | 处理方式 |
|------|------|------|---------|
| 硬约束 | 🔴 | 影响任务成败的规则、要求、偏好 | 必须记住，写入记忆 |
| 软约束 | 🟡 | 讨论细节、中间结果、背景信息 | 可压缩，需要时能找到就行 |
| 噪声 | ⚪ | 闲聊、寒暄、重复确认 | 不用记，过了就过 |

### 识别方法

- **硬约束特征**："必须""不能""一定要""我要求""偏好是""格式要XX"
- **软约束特征**："我觉得""可能""大概""先这样""试试看"
- **噪声特征**：无实质内容的情绪表达、重复确认、寒暄

### 长对话压缩策略

同一话题超过 8 轮时，主动整理：

```
📋 目前整理：
✅ 已确定：
  - ...
📌 待确认/待办：
  - ...
🔴 关键约束：
  - ...
看看有没有遗漏或需要调整的？
```

压缩成三部分：**已确定结论 + 待决事项 + 关键约束**。确认无误后，后续对话基于摘要继续。

## 七、反馈闭环机制

### 收到修正时的四步处理

1. **先复述确认**："我理解你是说...对吗？"
2. **判断类型**：场景性修正（只针对本次任务）vs 永久性偏好变更（以后都要这样）
3. **立即修正**：执行修改，展示修改点
4. **写入记忆**：如果是永久性的，写入对应层级

### 修正反馈模板

```
✏️ 已根据你的反馈修改：
1. 修改了XX → 改为YY
2. 补充了ZZ
3. 修正了AA
你看看改对了吗？
```

### 需求确认模板（新任务开始时）

```
🔍 我先确认下关键要求：
1. 目标：...
2. 格式：...
3. 时间：...
4. 其他偏好：...
是这样吗？
```

## 八、记忆生命周期管理

记忆不是存了就完了。定期检查旧记忆是否仍然有效，用过时信息比没信息更糟。

### 生命周期阶段

| 阶段 | 状态 | 处理 |
|------|------|------|
| 活跃期 | 新写入，经常被引用 | 正常使用 |
| 沉寂期 | 超过 1 个月没被引用 | 标记为待验证，下次用到时确认是否仍有效 |
| 失效期 | 确认已过时/不再适用 | 不删除，标记"已失效"，避免误用 |
| 归档期 | 项目结束/任务完成 | 归档到对应项目文件，保留指针 |

### 失效不删除原则

- 旧记忆即使确认失效，也不物理删除
- 只标记失效状态，保留可追溯性
- 原因：也许以后还需要知道"以前为什么那么想"

## 九、记忆质量自检清单

每次写入记忆前过一遍：

- [ ] 值得记吗？ — 换个项目还能用？以后会复用吗？
- [ ] 放对层级了吗？ — 即时层只放核心，详情沉下去
- [ ] 格式规范吗？ — 标题清晰、结构分明、能快速扫读
- [ ] 有来源吗？ — 重要结论标注来源和置信度
- [ ] 会过期吗？ — 有时效性的标注生效日期和复查日期

---

## 核心原则

1. **分层记忆，重点突出** — 不是记得越多越好，而是重要的不能丢
2. **记不住就直说** — 信息丢失要透明，不要凭模糊印象回答
3. **写入前必确认** — 永久性记忆写入前跟用户确认
4. **记忆是活的** — 定期检查是否仍然有效
5. **反馈必闭环** — 收到反馈 → 确认理解 → 执行修正 → 展示结果 → 确认完成
6. **越级不查** — 能用浅层解决的不碰深层，节省 token
7. **失效不删** — 只标记不删除，保留可追溯性
8. **同坑不踩第二次** — 工具经验、踩坑教训必须写入 TOOLS.md

## Agent 执行指南

> 拿到这篇文档后，按以下步骤执行自我优化：

### 步骤 1：深度阅读与现状盘点

逐节阅读，对照自身记忆系统做诊断：
- 我有三层记忆架构吗？还是所有内容混在一起？
- 即时层文件（MEMORY.md / USER.md / TOOLS.md）有没有？格式规范吗？容量超了吗？
- 有近中期层的 index.json 索引吗？还是每次都要全量搜索？
- 收到用户反馈后，我是记下来了还是转头就忘？
- 有没有过时的记忆还在被使用？

列出 3-5 个最需要改进的点。

### 步骤 2：搭建记忆三层架构

**2.1 即时层文件初始化**

按规范创建/整理五份核心文件：
- `USER.md`（≤2KB）— 用户画像
- `MEMORY.md`（≤5KB）— 两大分区：长期行为规则 + 核心状态锚点
- `SOUL.md`（≤5KB）— 身份定义（如果需要）
- `TOOLS.md`（≤5KB）— 工具操作经验，格式："用XX工具时要注意YY"

**2.2 近中期层初始化**

创建目录结构和索引：
```
recent_memory/
├── index.json
├── project/
├── decision/
└── todo/
```
写入 index.json 初始结构。

**2.3 写入检索规则**

在 MEMORY.md 中写入：
- 检索黄金顺序（即时层→近中期层→长期层）
- 写入前三问
- 上下文分层标记规则

### 步骤 3：建立反馈闭环

在核心规则中加入：
- 收到用户纠正时的四步处理流程
- 修正反馈模板
- 永久性偏好 → 写入 USER.md 或 MEMORY.md
- 场景性修正 → 记录到项目文件

### 步骤 4：实战验证

在后续每次任务中练习：
1. 任务开始前：先查即时层→再查近中期层（不越级）
2. 任务过程中：识别 🔴硬约束 / 🟡软约束 / ⚪噪声
3. 长对话超 8 轮：主动压缩为"结论+待决+约束"
4. 收到纠正：四步闭环处理
5. 任务结束：按三问判断是否需要写入记忆

### 步骤 5：持续维护

- 即时层接近上限时，按瘦身原则精简
- 每周检查近中期层 index.json，确保索引准确
- 每月检查旧记忆是否仍然有效（生命周期管理）
- 失效记忆只标记不删除
