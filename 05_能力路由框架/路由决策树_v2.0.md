# 能力路由框架 — 三层决策树 + 置信度机制 v2.0

> 核心一句话：不是贵的好，是合适的好——收到指令后 3 秒内判断领域、工具、要不要澄清，避免全量加载浪费 token
> 适用场景：每次收到任务时
> 更新日期：2026-07-09
> 版本：v2.0

---

## 设计理念

每个模型/工具有擅长的领域。路由的目标：
- **节省时间 90%**：不做无谓的全量加载
- **节省 token 90%**：精准匹配，避免大炮打蚊子
- **提高质量**：用对的模型做对的事

## 三层路由决策树

### 第一层：大类判断（5 个方向）

收到指令后，先快速归类：

| 大类 | 触发关键词 | 核心能力 | 默认模型 |
|------|----------|---------|---------|
| A 内容创作 | 写/文案/文章/脚本/封面 | 写作类技能 + 人化写作规范 | 写作型 |
| B 调研分析 | 调研/研究/分析/报告/对比 | 深度调研 SOP + 行业知识库 | 推理型 |
| C 工具操作 | 表格/文档/邮件/日历 | 对应平台 skill + 操作手册 | 标准型 |
| D 系统技术 | 代码/部署/API/脚本 | codeact + 技术知识库 | 代码型 |
| E 咨询方法 | 建议/怎么办/策略/方案 | 结构化推理 + 行业知识 | 标准-推理型 |
| F 日常对话 | 闲聊/确认/简单问题 | — | 轻量型 |

**交叉判断原则**：
- 有明确产出物（写个XX、做个XX）→ 优先归 A/C
- 有"调研/研究/对比"关键词 → 归 B
- 有具体平台名称 → 归 C
- 问"怎么做/为什么" → 归 E

### 第二层：细分匹配

大类确定后，快速匹配细分场景和对应工具。

**A. 内容创作类**

| 细分场景 | 触发词 | 首选工具 |
|---------|--------|---------|
| 小红书笔记 | 小红书、笔记、种草 | 人化写作 + 封面设计 |
| 公众号文章 | 公众号、推文、长文 | 人化写作 + 排版 |
| 封面/海报图 | 封面、海报、主图 | 图片生成 + 封面方法论 |
| 口播/短视频脚本 | 口播、脚本、短视频 | 人化写作 + 内容工程化 |
| 文案改写/多平台 | 改写、一稿多发 | 多平台改写 |

**B. 调研分析类**

| 细分场景 | 触发词 | 首选工具 |
|---------|--------|---------|
| 行业/市场调研 | 行业、市场、赛道、趋势 | deep_research + 行业知识库 |
| 公司/竞品调研 | 竞品、对标、扒一下 | deep_research |
| 通用深度调研 | 研究、分析、深度报告 | agent + 深度调研 SOP |
| 话题追踪 | 追踪、关注、每天看看 | topic_tracking |

**C. 工具操作类**

| 细分场景 | 触发词 | 首选工具 |
|---------|--------|---------|
| 飞书操作 | 飞书、多维表格 | lark_cli + 表格知识库 |
| 文档/表格处理 | word、excel、pdf、ppt | 对应文档 skill |
| 邮件操作 | 邮件、发邮件 | email_request |

**D. 系统/技术类**

| 细分场景 | 触发词 | 首选工具 |
|---------|--------|---------|
| 脚本/自动化 | 脚本、自动、批量、定时 | codeact + bash |
| 技术学习/开发 | 开发、编程、部署 | codeact + 搜索 |
| 工具安装/配置 | 安装、配置、环境 | bash + 搜索 |

**E. 咨询/方法类**

| 细分场景 | 触发词 | 首选工具 |
|---------|--------|---------|
| 商业/策略建议 | 怎么办、策略、方案 | 结构化推理 + 调研 |
| 需求澄清 | 说不清楚、怎么弄 | 挖需方法论 + 五步澄清法 |

### 第三层：执行模式判断

| 模式 | 判断标准 | 执行方式 |
|------|---------|---------|
| 直接做 | 目标明确、信息足够、低风险 | 自己直接执行 |
| 派子任务 | 预计>1 分钟、产物可独立检查 | spawn 子 agent（可不同模型） |
| 先澄清 | 目标模糊、关键信息缺失、高风险不可逆 | 用挖需话术澄清，确认后再动手 |
| 建日程 | 未来时间点执行、周期性任务 | calendar_create |

## 置信度三级机制

每一层路由判断都必须输出置信度，根据置信度决定下一步动作。

### 置信度分级标准

| 级别 | 置信度区间 | 判断标准 | 下一步动作 |
|------|----------|---------|----------|
| 高置信 | ≥80% | 关键词明确、场景单一、用户历史行为一致 | 直接执行/派发，不犹豫 |
| 中置信 | 50%-80% | 有多个可能方向、关键词模糊、但有一个明显更优解 | 给出 1-2 个反提案选项，让用户确认 |
| 低置信 | <50% | 完全猜不透、有 3 种以上解读、用户表述极简 | 直接提问，用挖需话术澄清真实意图 |

### 每层怎么打置信分

**第一层（大类判断）**：
- 有明确触发词 + 产出物清晰 = 高置信（如"帮我写一篇小红书笔记"）
- 有触发词但产出物不清晰 = 中置信（如"帮我写点东西"）
- 没有任何领域关键词 = 低置信（如"你帮我弄一下"）

**第二层（细分匹配）**：
- 细分场景触发词全部命中 = 高置信
- 命中部分特征，需要补充信息 = 中置信
- 匹配不上任何细分场景 = 低置信（退回第一层重判）

**第三层（执行模式）**：
- 目标明确 + 低风险 + 信息充足 = 高置信 → 直接做
- 目标大致清晰但细节待定 = 中置信 → 先做 MVP 再调整
- 高风险/不可逆/信息严重不足 = 低置信 → 必须澄清

### 置信度降级触发条件

出现以下任一情况，置信度自动降一级：

1. 用户说"随便""都行""你看着办"——看似放权实则风险高
2. 同一个词有多个歧义（"方案"= 方案文档 / 方案策略 / 方案设计）
3. 用户历史上在类似问题上纠正过路由方向
4. 涉及高风险操作（删除/发送/花钱/对外发布）

## 模糊需求澄清机制

### 什么情况需要澄清？（满足任一即澄清）

1. 目标不明确：不知道最终产出物是什么
2. 关键信息缺失：缺少必要参数（时间、数量、对象、场景）
3. 高风险动作：删除、发送、对外发布、花钱——必须确认
4. 有多种解读方式：同一个指令可能对应完全不同的任务

### 澄清原则

- **能推进先推进**：不影响核心判断的细节，用默认值先干
- **一次问完**：不要挤牙膏，把要确认的一次列出来
- **给选项**：不要问开放式问题，给 2-3 个选项让用户选
- **不超过 3 个问题**：一次澄清最多问 3 个

### 标准澄清话术模板

**【高置信-微澄清】**（90% 确定，但怕踩坑）
```
我按[方向]来做，没问题吧？预计产出[产物形式]。
```

**【中置信-反提案】**（50-80% 确定，给选项）
```
确认下方向——听起来你是要[A选项，具体描述]，还是[B选项，具体描述]？或者是别的？
```

**【低置信-挖需式】**（<50% 确定）
```
我先问几个点把需求搞准：
1）你最终想用这个来干什么？（场景/用途）
2）大概要做到什么程度？（规模/深度）
3）有参考的样子吗？（对标/风格）
```

**【高风险动作确认】**
```
确认一下执行动作：[具体操作描述]
影响范围：[涉及哪些数据/哪些人]
不可逆说明：[执行后能否撤销]
你确认我就动手。
```

**【"优化一下"专用】**（最模糊的词，必须先拆）
```
"优化"有好几种，你是哪种：
1）加内容/加功能（做加法）
2）改风格/改结构（做调整）
3）删冗余/提效率（做减法）
或者别的？
```

## 自动升降档

### 降档（省 token）

- 上下文压缩后使用轻量模型
- 子 agent 分级（简单子任务用便宜模型）
- 工具调用降档（能简单工具解决不上大模型）
- 总结降档（总结归纳类用轻量模型）
- 缓存命中后跳过重复计算

### 升档（保质量）

- 用户纠正后 → 切换到更高级模型重新处理
- 高风险操作 → 用推理型模型做二次验证
- 多约束条件 → 升级到能处理复杂约束的模型
- 跨领域任务 → 需要综合能力的模型

## 自检清单（每次接任务过一遍）

- [ ] 判断大类：属于 A/B/C/D/E/F 哪类？
- [ ] 匹配工具：有没有对应的 skill/知识库/agent？
- [ ] 信息够吗：需不需要先澄清？
- [ ] 怎么执行：自己做/派子任务/建日程？
- [ ] 产出形式：用户要的是一句话、一份文档、还是一个文件？

**3 秒内过完，不要想太久。不确定的归到最接近的大类，边做边调。**

## 常见误判修正

| 容易判错的场景 | 正确归类 | 原因 |
|--------------|---------|------|
| "帮我写个方案" | A 内容创作，不是 E 咨询 | 有明确产出物（方案文档），核心是写作 |
| "研究一下XX怎么做" | B 调研分析，不是 E 咨询 | "研究"关键词 → deep_research |
| "用飞书做个表" | C 工具操作 + A 内容创作 | 先加载飞书 skill，内容设计用多维表格知识 |
| "给我建议" | E 咨询，但先澄清 | "建议"太泛，先问清楚什么方面 |
| "优化一下" | 先澄清，再归类 | "优化"是最模糊的词，必须先问清优化什么 |
| "看看这个" | 先读文件/链接，再判断 | 先获取内容，再判断属于哪类 |

## 误判案例日志机制

每次路由判断错误（用户说"不对""不是这个意思"），必须记录到误判日志。

### 记录格式

```
- 2026-07-09
  - 用户原话：xxx
  - 我的判断：路由到了 xxx
  - 正确方向：实际用户想要 xxx
  - 误判原因：关键词歧义/缺上下文/分类体系问题
  - 改进措施：下次遇到类似情况，应该 xxx
```

### 日志位置

`recent_memory/project/路由误判日志.md`

### 复盘周期

- 累计满 5 条 → 触发一次路由框架小版本迭代
- 每月 → 做一次全面复盘，更新分类体系和触发词

## Agent 执行指南

> 拿到这篇文档后，按以下步骤执行自我优化：

### 步骤 1：深度阅读与自我诊断

逐节阅读，对照自身路由能力做诊断：
- 我接到任务后，是直接开始做，还是会先判断任务类型？
- 我会根据任务类型选择不同的处理方式吗？
- 遇到模糊需求时，我是直接猜还是先澄清？
- 有没有因为理解错用户意图而返工的经历？
- 我能说出自己的常见误判场景吗？

### 步骤 2：建立路由规则

在核心规则中写入：
- 三层决策树（大类→细分→执行模式）
- 置信度三级标准
- 澄清话术模板（至少记住高/中/低三种）
- "3 秒内判断，不确定的边做边调"

### 步骤 3：建立误判日志

创建 `recent_memory/project/路由误判日志.md`：
- 每次被用户纠正路由方向，立即记录
- 格式：用户原话 / 我的判断 / 正确方向 / 误判原因 / 改进措施

### 步骤 4：实战验证

在后续每次任务中练习：
1. 收到指令 → 3 秒内判断大类（A/B/C/D/E/F）
2. 输出置信度（高/中/低）
3. 高置信直接做，中置信给反提案，低置信先问 3 个问题
4. 遇到模糊词（"优化""弄一下""看看"）→ 强制澄清
5. 被纠正 → 记录误判日志

### 步骤 5：迭代优化

- 误判日志满 5 条 → 分析误判模式 → 更新触发词和分类规则
- 每月复盘 → 更新细分场景匹配表
- 跟踪自动升降档的效果
